這個週末，在脆上看到有人用 vibe coding 做了一個興趣記錄 app，做得不錯，是我一直想做的app，但它讓我想到一件大多數 vibe coding 新手從沒考慮過的事，對我們產品開發者來說，如同肌肉記憶般的事。

### Vibe Coding的從想法到產品

Vibe coding 讓「從想法到產品」這件事變得離譜地簡單。你描述你要什麼，AI 幫你寫程式，突然間你就有了一個 app。以前要做好幾個月的東西，現在一個週末就能搞定。

當然，很多這類專案都是「重造輪子」——已經有人做過的東西，自己再從頭做一次——記帳、行程管理系統、訂單系統⋯⋯

我完全沒意見。我自己也這樣。你想做自己的輪子？請便。這本來就是做產品最好玩的地方。

但危險的地方在這裡：當你加上訂閱制，開始接真實用戶的那一刻。

### 洞察

我認為 vibe coding 新手最大的坑，不是程式碼品質，而是**商業模式的幻覺**，也就是以為做出來、掛上訂閱，收入就會來。

其實如果只是想分擔伺服器成本，我認為放個贊助按鈕就夠了。不需要為一個週末做出來、三個月後可能就不維護的東西，收人家每月 150 塊。

但真正的問題比定價更深。關鍵是：**你的產品替用戶保管了什麼。**

我認為資料責任有一個光譜：

- **短期、低依附的資料**——計時器、本週的追蹤器、心情日誌。App 死了，用戶聳聳肩，換一個就好。沒什麼大不了。
- **長期、高依附的資料**——開發日誌、編織作品紀錄、毛線資料庫、日記、記帳。這些是用戶花了好幾個月甚至好幾年累積的東西。它們是個人檔案，是記憶本身。

如果做的是後者，我認為有一個問題值得在**寫下第一個 prompt 之前**就先想清楚：

*當你不維護了，用戶的資料怎麼辦？*

我自己就有好幾次經驗，用得順手的記帳軟體寄來一封文辭並茂的信，告訴我很抱歉，我們決定不經營了。或者日記軟體突然要求要從買斷方案升級成訂閱方案，否則就拿不回原本的日記紀錄，我四五年以上的日記就在升級的過程中因為某些不知名的原因，消失在雲端某處。

因為大多數 side project 都會死。這不是道德缺陷，是統計事實。我認為真正的失敗，是根本沒想過這件事。

### 為什麼重要？

當一個存了長期資料的 vibe coding 產品停擺，開發者失去的是一個興趣專案。用戶失去的，是他們託付給你的好幾年的紀錄。

如果做的東西會累積用戶的記憶，以下是我自己會在第一天就想的事：

- 用戶能不能用標準格式匯出資料？
- 這些資料能不能搬到別的工具？
- 有沒有跟用戶說清楚，這是個人專案，沒有 SLA？

我不認為我們需要在上線前全部解決。但我認為至少要**想過**。我認為標準不是「做出企業級產品」，而是「別不小心把別人的記憶綁架了」。

至於我的話，因為我很有自知之明，所以開產品的時候，我從來就沒打算幫用戶存什麼東西。把他們要的東西寄到他們信箱，結束 :P
